# i18n

Engine 把 i18n 拆成**两层**：

1. **URL 路由层** —— `/zh/about` vs `/en/about` 怎么解析、怎么生成。无 React 依赖，server + client 通用。
2. **翻译消息加载层** —— 字典从哪儿来、怎么缓存、怎么协商。由 SiteHooks `intl` 配置统一收口。

两层独立，可以只用一层。默认推荐使用 engine 内置的请求作用域 `getI18n()`；如果业务已经有 react-i18next / lingui / formatjs，也可以把 engine 下发的 `intl.messages` 转接过去。

## URL 路由层

### `parseLocale(pathname, config)`

从 URL 拆 locale + 干净的业务 pathname。

```ts
import { parseLocale } from '@novel-isr/engine/runtime';

const cfg = { locales: ['zh', 'en'], defaultLocale: 'zh' };

parseLocale('/en/books/1', cfg);
// → { locale: 'en', pathname: '/books/1', hasPrefix: true }

parseLocale('/books/1', cfg);
// → { locale: 'zh', pathname: '/books/1', hasPrefix: false }
```

业务路由表只需要匹配 pathname，不用关心 locale 前缀。

### `withLocale(pathname, locale, config)`

生成可跳转的带 locale 前缀 URL。

```ts
withLocale('/books/1', 'en', cfg);   // → '/en/books/1'
withLocale('/books/1', 'zh', cfg);   // → '/books/1'  （默认 locale 不前缀）

withLocale('/books/1', 'zh', { ...cfg, prefixDefault: true });
// → '/zh/books/1'
```

适合 Header 的语言切换链接。

### `negotiateLocale(acceptLanguage, config)`

按 Accept-Language 头协商（RFC 4647 lookup）。

```ts
negotiateLocale('zh-CN,en;q=0.9', cfg);   // → 'zh' （主语言匹配）
negotiateLocale(null, cfg);                // → defaultLocale
```

### `alternates(pathname, config)`

生成 `<link rel="alternate" hreflang>` 列表。

```ts
alternates('/about', cfg);
// → [
//     { hreflang: 'zh', href: '/about' },
//     { hreflang: 'en', href: '/en/about' },
//   ]
```

直接喂给 SiteHooks 的 `seo.*.alternates` 字段。

### `resolveI18nConfig(intl?)`

从 SiteHooks 的 `intl` 字段提取 URL 路由配置并补默认值。消除消费者侧那段「从 `siteHooks.intl` 抠出 `locales`/`defaultLocale`/`prefixDefault` 再补默认」的样板。

```ts
import { resolveI18nConfig } from '@novel-isr/engine/runtime';
import siteHooks from './entry.server';

const I18N = resolveI18nConfig(siteHooks.intl);
// → { locales, defaultLocale, prefixDefault }
```

行为：
- 缺省 → `{ locales: ['en'], defaultLocale: 'en' }`
- `locales` 给了但 `defaultLocale` 没给 → 取 `locales[0]`
- 多余字段（`endpoint` / `load` / `ttl` / `detect`）忽略

## 翻译消息加载层

在 `ssr.config.ts runtime.i18n` 里声明：

```ts
import fallbackLocal from './src/config/site-fallback-local.json';

runtime: {
  i18n: {
    locales: fallbackLocal.site.locales,
    defaultLocale: fallbackLocal.site.defaultLocale,
    endpoint: '/api/i18n/{locale}/manifest',
    fallbackLocal: fallbackLocal.i18n.strings,
    // 相对 endpoint 统一拼到 runtime.services.api；不再单独配置独立 i18n service
    timeoutMs: 1200,
    ttl: 60_000,
  },
}
```

纯本地站点也可以不配置 `endpoint`，只保留 `fallbackLocal`：

```ts
runtime: {
  i18n: {
    locales: ['zh', 'en'],
    defaultLocale: 'zh',
    fallbackLocal: {
      zh: { 'home.title': '首页' },
      en: { 'home.title': 'Home' },
    },
  },
}
```

Engine 自动：
- 按请求协商 locale（cookie `locale` → Accept-Language → defaultLocale）
- 用 `runtime.services.api` 拼接远端地址
- LRU + TTL + 并发去重 + SWR + 失败兜底本地字典
- 把结果作为 `intl` prop 传给 `<App/>`，同时随 RSC payload 到客户端
- 在 server render 期间注册请求作用域，业务可以直接调用 `getI18n()`

进阶项目仍可用 `defineSiteHooks({ intl })` 自定义 loader：

```ts
intl: {
  locales: ['zh', 'en'],
  defaultLocale: 'zh',
  load: async locale => ({ locale, messages: await loadMessages(locale), direction: 'ltr' }),
  ttl: 60_000,
}
```
- 自动设 `Content-Language` 响应头

### 业务侧消费：`getI18n`

```tsx
import { getI18n, getI18nLocale } from '@novel-isr/engine/runtime';

getI18n('home.hero.title');
getI18n('book.count', { count: 12 }); // 字典里写 "共 {count} 本书"
getI18nLocale('zh-CN');
```

变量占位符是 `{name}`，不是 `$name`：

```json
{
  "book": {
    "count": "共 {count} 本书",
    "rating": "{title} 评分 {score}"
  }
}
```

`fallback` 是第三个参数；不传时返回 key 本身，便于开发期暴露缺失字典：

```ts
getI18n('missing.key', undefined, '兜底文案');
```

`IntlPayload` 结构：

```ts
interface IntlPayload {
  locale: string;
  messages: Record<string, unknown>;
  direction: 'ltr' | 'rtl';   // 自动按 locale 推断（ar/he/fa/ur → rtl）
  source?: string;            // 诊断字段，如 'remote' / 'local-fallback'
}
```

`source` 不参与翻译查找，只用于诊断。Engine 会把它写入 `X-I18n-Source` 响应头，
开发态 Novel ISR Inspector 也会展示它。业务不需要手写 fetch、超时、dotted key 展开：

```ts
// 远端命中：x-i18n-source: remote
// 本地兜底：x-i18n-source: local-fallback
```

## SSR / CSR recovery 行为

**SSR + RSC 流水线下的两端一致消费**：

1. Engine 在 server 端 fetch 字典 → 注入 RSC payload
2. 客户端 hydrate 时拿到同一份，不需要二次 fetch
3. Server Component / Client Component 都通过 `getI18n()` 读当前 payload
4. 客户端导航拉 `_.rsc`，payload 带 `intl`，engine 自动更新客户端 i18n store

CSR recovery 分两种：

- 默认 RSC shell fallback：浏览器先 fetch 当前页面 `_.rsc`，拿到 `intl` 后再渲染页面，仍然使用 `getI18n()`。
- 业务不需要维护自定义 CSR App 或第二套路由；server/RSC 不可用时由 engine 统一展示最终不可用壳。
- 如果服务端完全不可用且 `_.rsc` 也失败，只能显示最终不可用壳；这时远程 i18n 数据源不可达，不会再假装有业务文案。

判断当前是不是远端下发，直接看响应头：

```bash
curl -sS -D - http://localhost:3000/ -o /dev/null | grep -i 'content-language\\|x-i18n-source'
# content-language: zh
# x-i18n-source: remote
```

## URL 重写 / 重定向

`parseLocale` / `withLocale` 只负责解析和生成 URL，不隐式跳转。需要把 `/about` 规范化为 `/zh/about` 时，应该在业务自己的边缘层、网关或显式 middleware 里做 301/302；engine 不会在 `beforeRequest` 返回值里识别私有 `__redirect` 字段。
